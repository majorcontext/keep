# Keep v1.0.0 Design

**Date:** 2026-04-03
**Status:** Approved
**Goal:** Public open-source release with semver stability guarantees

## Context

Keep is at v0.2.0 with all core components implemented: CEL-based policy engine, YAML rule files, MCP relay, LLM gateway, CLI, Go library API, and Anthropic codec. The codebase has 56 test files with 66-100% coverage across packages, comprehensive documentation, and production features (graceful shutdown, hot reload, streaming, secret detection, rate limiting).

v1.0.0 graduates Keep to a public release. The public Go API surface, rule file format, CLI interface, and config schemas all become semver-stable contracts. Breaking changes are acceptable before 1.0 but not after.

## Positioning

Keep is a standalone library and set of small composable tools. Moat is the flagship deployment context, but Keep works independently — someone with no Moat context can install it, write rules, and run it.

## What Ships in v1.0.0

### 1. New Provider Codecs

**OpenAI/Codex codec (`llm/openai/`)**

Implements the existing `llm.Codec` interface for the OpenAI Chat Completions API.

- Decomposes requests/responses into `keep.Call` objects
- Content block types: `tool_use`, `tool_result`, `text`, `image_url`
- Streaming support via SSE (same pattern as Anthropic codec)
- Function calling (legacy) mapped to `tool_use` canonical form
- Gateway gains `provider: openai` config option

**Gemini codec (`llm/gemini/`)**

Implements `llm.Codec` for the Gemini `generateContent` API.

- Decomposes requests/responses into `keep.Call` objects
- Part types: `functionCall`, `functionResponse`, `text`, `inlineData`
- Streaming via Gemini's SSE format
- Gateway gains `provider: gemini` config option

**Design principle:** Each codec is a self-contained package. No shared base class — just the `Codec` interface. Provider-specific types stay in their own package (e.g., `openai.Request`, `gemini.Part`).

### 2. Canonical Operation Taxonomy

A normalization layer that maps provider-specific operation names to canonical names, so rules can be written once and match across providers and MCP servers.

**Canonical categories:**

| Category | Canonical Operations |
|----------|---------------------|
| `file` | `file.read`, `file.write`, `file.list`, `file.delete` |
| `code` | `code.execute`, `code.analyze` |
| `net` | `net.http`, `net.dns`, `net.connect` |
| `data` | `data.query`, `data.mutate`, `data.schema` |
| `msg` | `msg.send`, `msg.read`, `msg.list` |
| `auth` | `auth.token`, `auth.credential` |
| `shell` | `shell.exec`, `shell.spawn` |

This is a starting vocabulary, not a closed set. New canonical names can be added in 1.x without breaking changes. The taxonomy is intentionally shallow (two levels) to stay learnable.

**How mapping works:**

```yaml
# Built-in mappings ship with each codec and relay
# Users can extend/override in rule files
normalize:
  mappings:
    read_file: file.read
    readFile: file.read
    ReadFileTool: file.read
    execute_command: shell.exec
    run_bash: shell.exec
```

**Rule matching behavior:**

- Rules can use raw names (`read_file`) or canonical names (`file.read`)
- When `normalize: true` on a scope, the engine checks both the raw and canonical name
- No forced migration — existing rules continue to work untouched
- Canonical names in rules match any raw name that maps to them

**Where mappings live:**

- Built-in defaults ship per codec/provider (e.g., common MCP server tools, Anthropic computer-use tools)
- User-defined mappings in rule files override built-ins
- The CLI `keep validate` warns on unmapped operations (info-level, not error)

### 3. API Surface Audit & Cleanup

Review and finalize all public API before stability guarantee.

**`keep.go` top-level API:**

- Audit exported types: `Engine`, `Call`, `Decision`, `Mutation`, `AuditEntry`, `RuleSet`
- Audit exported functions: `Load`, `LoadFromBytes`, `Evaluate`, `Reload`, `ApplyMutations`, `ValidateRuleBytes`
- Review option types: `WithAuditHook`, `WithRateStore`
- Ensure naming consistency across the surface
- Remove anything experimental or under-baked

**`llm/` package API:**

- `Codec` interface — review method signatures
- `EvaluateRequest`, `EvaluateResponse`, `EvaluateStream` — confirm naming and error semantics
- `DecomposeConfig` — review fields

**Rule file format:**

- Lock down YAML schema: scopes, rules, actions, when expressions, normalize block
- Document what's stable vs experimental
- Version field in rule files (default `v1`, future-proofing)

**Config formats:**

- `relay.yaml` schema finalized
- `gateway.yaml` schema finalized (with new `provider` options)

**CLI interface:**

- `keep validate` and `keep test` — stable
- `keep normalize` (new) — show canonical mappings for a rule set
- Version output includes build metadata

### 6. LLM-as-Judge (POC)

A proof-of-concept for delegating policy decisions to an LLM when deterministic CEL expressions aren't sufficient. For example, evaluating whether content is appropriate, whether a tool call is consistent with stated intent, or whether a response contains hallucinated claims.

**Scope for v1.0.0:** POC only. The goal is to prove out the integration pattern and surface area, not to ship a production-grade judge. Exact design (CEL function like `judge()`, separate rule action, standalone evaluation mode) to be brainstormed during implementation.

**Key questions to resolve during implementation:**

- How does the judge integrate with the rule evaluation pipeline? (CEL function, post-eval hook, separate action type)
- What LLM providers are supported for judging? (reuse existing codecs or separate config)
- How are judge prompts specified and versioned?
- Latency implications — judge calls are async/slow vs. synchronous eval
- Caching and cost controls
- How results feed back into the decision (hard deny, score threshold, advisory)

**What ships:** A working integration that demonstrates the pattern, with at least one end-to-end test. Explicitly marked as experimental/unstable — not covered by semver guarantees until a future release.

### 7. Testing & Stability

**New test coverage:**

- OpenAI codec: unit tests + end-to-end pipeline tests (same pattern as Anthropic)
- Gemini codec: unit tests + end-to-end pipeline tests
- Normalization: unit tests for mapping resolution, canonical matching, override behavior
- Cross-provider: fixture-based tests showing same rules evaluated across all three codecs

**Hardening:**

- Fuzz testing for rule parsing and CEL compilation
- Benchmark suite for engine evaluation (establish performance baseline before 1.0)
- `cmd/` binaries get basic smoke tests (startup, graceful shutdown, config validation)
- Audit package coverage improved from 33% to >70%

## Out of Scope (1.x or Later)

- MCP resources/prompts/sampling support
- Plugin/extension system
- Remote rule storage
- Additional LLM providers beyond the big three
- Web UI / dashboard
